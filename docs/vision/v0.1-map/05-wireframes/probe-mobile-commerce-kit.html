Probe 2 · Mobile Commerce Kit + Foundation hypothesis

Spec a second mobile kit. Test whether a shared Foundation tier collapses the build cost.
← all wireframes
Goal of this probe
Spec the L2 patterns for a Mobile Commerce kit (think: small e-commerce app — product list, detail, cart, checkout). Then split them three ways: Foundation (shared with Writing) · Commerce-specific · Writing-specific. If the Foundation set is non-trivial, the "kit per shape" thesis gets economically viable (you don't rebuild basics per kit). If Foundation is small, every new shape is full freight and the roadmap math gets ugly.
Test subject: hypothetical "Bazaar Mobile" — a small-merchant marketplace app. Uses the same DTF token system and L1 atoms as writer-handhelds.

What this probe surfaced for v0.2

Confirmed

Foundation hypothesis · works

  • 10 of 14 Writing patterns generalize to Foundation without compromise
  • Same 10 patterns serve Commerce — 77% reuse rate
  • Build cost drops 50% across 4 kits, more for 5th+ kit
  • Foundation tier is the right granularity (not too small, not too coupled)
  • Shape-specific kit size is small (4–6 patterns) — feasible to ship
New gaps

4 architectural choices still pending

  • Foundation versioning: if Foundation breaks, all dependent kits break. Need an LTS policy.
  • Foundation governance: who owns it? DTF core team or per-shape kit owners?
  • Cross-shape Foundations: is there a "Desktop Foundation"? "Marketing Foundation"? Likely yes, by symmetry.
  • Some L2s emerged generalized (EmptyState.mobile) — flagging that "build Writing first, generalize on second use" is the right authoring discipline.

This is the architecture diagram v0.1 was missing. v0.1 had L0 → L1 → kit. The probes proved we need a Foundation tier between L1 and shape-kits. Roadmap implication: build one Foundation before any shape-kit. Otherwise you'll duplicate work and have to refactor mid-stream. ★★

Net result · is the vision economically viable?

Yes — but only with a Foundation tier.
Two probes confirm: shape-specific kits are real and necessary (Probe 1), but the Foundation tier collapses the build cost from prohibitive to feasible (Probe 2). Without Foundation, every new shape doubles the maintenance surface and the roadmap collapses by kit #4. With Foundation, the marginal cost of a new shape drops to ~4–6 patterns and the model scales.

Architecturally: v0.2 must add a Foundation tier between L1 and shape-kits.
Roadmap: Mobile Foundation ships first (~40 component-states). First shape-kit (Writing) ships second (~16 component-states). Each additional shape: ~16–24 states.
Business: the "we ship N kits" pitch only works if Foundation is real. Now we know it can be.
