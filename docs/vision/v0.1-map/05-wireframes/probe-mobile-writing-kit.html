Goal of this probe
Spec the actual L2 archetypes that would ship in the "Mobile Writing" kit, with enough detail that we can ask: is this kit meaningfully different from the Dashboard kit? If yes → "kit per shape" thesis is real. If no → the L1 set alone is enough and we should simplify the vision.

First observation: overlap is ~30% conceptually (EmptyState ≈ NoteListEmpty, ConfirmDialog ≈ ConfirmSheet, SettingsRow ≈ SettingsRow.mobile) — but every overlap requires a different L2 because mobile patterns have different gestures, dimensions, and platform conventions. Conclusion so far: the thesis holds. 1
  • A Dashboard ConfirmDialog is a centered modal with overlay. A mobile ConfirmSheet slides up from bottom and respects safe-area inset. Same L1 (Button, Text) but completely different L2 composition. Can't share.

L1 coverage check · what we'd need before building this kit

exists Button, IconButton, Input, Label, Text, Checkbox, Switch, Toggle, Avatar, Badge, Toast, Tooltip, Menu, Select
need new L1 SwipeContainer (gesture wrapper exposing left/right action slots)
need new L1 BottomSheet (modal surface anchored to bottom with drag-handle, snap-points)
need new L1 SafeAreaInset (token-driven padding for notch / home indicator)
need new L1 TabBarItem (icon + label + badge slot, active state)
platform-specific NavigationBar.ios vs AppBar.android — back gesture differs, large-title behavior differs, title alignment differs
  • 4 new L1s needed before this kit can ship. This is a finding for v0.2: kits don't just compose existing L1 — they pull new L1 requirements upstream. The vision needs a "kit requested an L1" pathway.
  • Platform-specific L1s (iOS vs Android nav bar) are a separate category. Either the kit ships both, the kit ships one neutral version, or we accept platform forks. Open decision for v0.2.

Composed from 6 L2 patterns + 1 L1

  • AppShell.mobile · wraps everything · gives SafeAreaInset + tabbar slot
  • TopBar · variant=large-title · 2 IconButton actions
  • SearchHeader · variant=idle (collapsed pseudo-input)
  • NoteCard × 3 · one with pinned indicator slot filled
  • Button (L1) · variant=floating-action · positioned absolute (no L2 needed)
  • BottomTabBar · variant=4-tabs · "Notes" active
Zero off-system values. Every padding, color, and radius reads from a token in the writer-handhelds project. Zero new L2s invented to ship this screen. The kit covers it. This is the bar.

What we learned by building this

  • FAB doesn't need a dedicated L2. It's just Button with a floating-action variant + absolute positioning recipe. L1 + recipe
  • The "large title" navigation pattern is meaningfully different from desktop page header. Justifies TopBar as its own L2.
  • SearchHeader appears in 90% of mobile apps with similar conventions. Strong candidate for an L2 in a "Mobile Foundation" sub-kit shared across mobile-shape kits.
  • One screen used 6 L2s — meaning the per-screen overhead of composition is significant. The L2 kit must exist for mobile to be productive.

Probe conclusion · does "kit per shape" hold?

Confirmed

Yes — the thesis is real

  • Mobile Writing kit shares ~30% conceptually with Dashboard kit but ~0% in composition
  • "Same intent, different composition" is the actual story (ConfirmDialog vs ConfirmSheet)
  • Building real L3 screens without the L2 kit is painful — proves the kit's existence is necessary
  • L2 kits are bounded sets (~10–15 patterns), small enough to spec and ship per shape
But surfaced new gaps

5 things v0.1 didn't account for

  • Kits drive new L1 requirements upstream (SwipeContainer, BottomSheet, SafeAreaInset, TabBarItem)
  • Platform variants (iOS vs Android) are a real fork — need explicit policy
  • A "Mobile Foundation" shared sub-kit might be cleaner than duplicating across mobile shapes
  • L2 patterns have variants (NoteCard: default/pinned/selected) — wireframes treated L2s as monolithic
  • Recipes vs L2 distinction got tested (FAB = L1+recipe, not L2) — the rule held but needs sharper criteria

What this probe contributes to v0.2

#DecisionReasoning from this probe
1 Vision must show ≥2 kits side-by-side The thesis is invisible until you see Dashboard kit and Mobile Writing kit on the same page. v0.1 only showed one.
2 Add "Foundation kits" tier Mobile Foundation = the SwipeRow / SearchHeader / Toast.mobile / TabBarItem set shared across Writing, Reading, Commerce mobile kits. Saves duplication. New architectural concept v0.1 missed.
3 Document the L1-pull pathway Kits don't just consume L1, they request new ones. Vision needs to show this loop.
4 Make L2 variants first-class Every L2 has 3–6 variants. v0.1 wireframes treated them as singular. Inventory drift.
5 Adopt explicit platform-fork policy Either "kits are neutral, apps fork at integration" or "kits ship .ios + .android" — pick one in v0.2.
6 Sharpen the "L2 vs recipe" criterion Building FAB taught us: if it's just a styled L1 + positioning, it's a recipe, not an L2. Write the rule explicitly.
7 Estimate kit-build cost Mobile Writing kit = 14 L2s × ~4 variants each × Figma + code = ~56 component-states to ship. If we're promising N kits, the build cost is real and should be in the roadmap.
Net result of Path B: the vision's central thesis is confirmed by a concrete artifact, but the probe surfaced 7 new architectural concerns that v0.1 didn't address. v0.2 will be meaningfully better-shaped because we did this probe first instead of just refining v0.1's wording. ★
