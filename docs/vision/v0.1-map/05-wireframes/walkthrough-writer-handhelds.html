01 · Onboarding

Brand → archetype → working scaffold
holdsStep 1 · Brand color + ladder
Writer Handhelds went through this exact step. #286CE5 → 20-step ladder visible in primitives.css. Anchor mode worked.
bendsStep 1 · One brand color is not enough
Real project has 8 roles (brand, danger, warning, info, success, greyscale, desaturated, neutral). The wireframe shows only one swatch picker. In practice users need to pick at least brand + neutral + maybe accent. Refinement: Step 1 should be "pick brand" (anchor) then auto-derive the others with one-tap overrides, OR a multi-swatch UI.
missingTheme step (light + dark)
Writer Handhelds ships both themes with non-trivial differences (e.g. light on-component=fixed-white, dark also white but step indices differ). The wireframe never shows a "choose themes / configure dark mode" step. Refinement: Add Step 1.5 — "Themes: light only / dark only / both" — with a preview of how the chosen brand reads in both.
breaksStep 3 · Archetype grid (6 cards: Dashboard, Marketing, etc.)
The wireframe lists desktop-web shapes. There is no "mobile writing app" tile. Even "mobile app" is too coarse — a writing app needs editor toolbars + content cards; a chat app needs message bubbles + composer; a finance app needs charts + tile dashboards. Refinement: either expand the archetype list (12–20 shapes including Mobile-Reading, Mobile-Editor, Mobile-Feed, Mobile-Commerce) or add "Start blank + tell us what you're building" as fallback. The kit chosen here is the most consequential onboarding decision and the wireframe currently undersells it.
missingSurfaces config
Writer Handhelds has 8 surface levels. The wireframe doesn't mention surfaces at all — they're a configured artifact in config.json. Refinement: surfaces probably shouldn't be in onboarding (too advanced), but the output screen ("what you got") should mention them, and there needs to be a "Tune surfaces" entry point elsewhere.
bends"Open in Figma" preview
Showed a desktop file with 6 pages. For mobile project, file structure is probably the same — but the L2 Kit page would show mobile patterns. Refinement: the preview should be archetype-aware (show the actual kit they'd get), not a generic dashboard skeleton.

02 · Archetype Kit

The L2 kit + L3 examples + recipes
holdsSix-page Figma file structure
Tokens / L1 / L2 Kit / L3 Examples / Recipes / My Work — this structure is shape-agnostic and works for mobile.
breaksThe 11 L2 patterns shown (AppShell, StatTile, DataTable, FilterBar…)
None of these are right for Writer Handhelds. Mobile Writing kit would be roughly:
  • AppShell (mobile · status bar + content + bottom nav, not sidebar + content)
  • BottomNav (4–5 tabs)
  • TopBar (back / title / actions)
  • NoteCard (list item)
  • NoteListEmpty (empty state with illustration)
  • EditorCanvas (the writing surface)
  • EditorToolbar (formatting actions)
  • FormatSheet (bottom-sheet picker for fonts/sizes)
  • SearchBar (with cancel + suggestions)
  • SwipeRow (swipe-to-delete row)
  • ContextMenu (long-press)
  • SettingsRow (with disclosure / toggle / switch)
  • Confirm sheet (iOS action sheet style)
Refinement: the wireframe should make clear that the L2 grid is archetype-specific and show two different examples (Dashboard kit + Mobile Writing kit side-by-side) to drive the point that "kit per shape" is real.
bendsL3 example annotated (the "Users" page)
For writer-handhelds, equivalent L3 examples would be "NoteList screen" and "Editor screen" — both annotated showing how AppShell + BottomNav + NoteCard compose, or AppShell + EditorToolbar + EditorCanvas compose. Concept holds; specifics need swapping.
holdsPromises / doesn't promise split
Shape-agnostic. Works as drawn.
holdsRecipes vs L2 distinction
For mobile: "toolbar-with-overflow", "pull-to-refresh header", "keyboard-aware footer" are clearly recipes (compositions), not components. The split survives the mobile context.
missingPlatform variants inside the kit
Mobile means iOS conventions ≠ Android. Bottom-sheet behavior, back-gesture, action-sheet style all differ. The wireframe doesn't address whether the kit has iOS/Android variants or stays platform-neutral. Refinement: add a callout — does the kit ship one neutral set, or platform-tagged variants?

03 · Archetype Gallery

Archaeology catalog UI · 127 archetypes etc.
bends"Crawl the live product" assumption
Wireframe assumed Playwright + HTML extraction. Writer Handhelds is a native mobile app. No DOM to crawl. Archaeology pipeline needs a different ingestion path: screenshot-based vision (CLIP/GPT-4V), or instrumentation hooks via React Native/Flutter inspector, or design-time Figma file scan. Refinement: the wireframe's top header should show ingestion source ("HTML crawl" / "iOS build inspector" / "Figma library scan") — different sources have different fidelity tradeoffs.
holdsCatalog grid + fidelity score + filters
The visual concept (tiles, instance counts, on-system %) is platform-agnostic. Whatever the ingestion source, the gallery format works.
holdsDrift tab · property-by-property table
padding / radius / shadow / font-size comparison works as drawn. Even more useful on mobile where 1pt off-grid is visible at 2x/3x density.
holdsSpec tab + reviewer actions
LLM auto-drafts spec from cluster representatives; same model applies regardless of source platform.
missingTheme dimension in drift detection
Writer Handhelds has light + dark. If a UserRow drifts only in dark mode (e.g. wrong on-component step), the wireframe's drift table doesn't surface that. Refinement: drift report should have a theme column or per-theme stats.
missingDensity / viewport dimension
Mobile has device sizes (small iPhone, large iPhone, tablet). Same drift but at compact vs regular density. Wireframe currently shows "38 routes · 4 viewports" — for mobile this should be "device classes."

04 · Drift Dashboard

Exec rollup · 70% org-wide, trend chart, top sources
holdsOrg rollup, per-product cards, trend chart
Writer Handhelds would be one card in the per-product grid. Trend logic works. Exec slide format works.
bendsTop drift sources table · "Fix →" buttons
On the web product the "Fix →" button could plausibly open a codemod / PR. For native mobile, "fix" means filing a ticket or opening the Figma component (since CSS doesn't apply). Refinement: the action surface depends on ingestion source — needs to be explicit per row.
breaksPer-PR drift table
Wireframe shows GitHub PRs with Δ Fidelity per PR. For mobile apps with biweekly builds + native code, "per PR drift" is less useful than "per release drift" or "per design-spec drift." Refinement: the engineering view should adapt to the artifact cadence (PR for web, build/sprint for mobile, page for marketing sites).
missingWhere token-version drift shows up
Writer Handhelds shipped 17 token versions in ~16 days. Some components in the live product use v1.0.5 tokens; new screens use v1.0.10. The dashboard never shows "X% of your product is on stale tokens." Refinement: add a "token version distribution" chip — this is unique to systems with versioned tokens (which DTF is).

05 · Compose from Intent

AI compose in Figma · intent → real instances
holdsIntent input + context detection
"Note list with search bar and floating compose button" is a perfectly natural intent for writer-handhelds. Context auto-detection (project = Writer Handhelds, theme, page, available kit) all applies.
bendsExample shown was a desktop login form
Not a problem in itself — but the wireframe leaves the impression that "compose" is web-shape. We should show 3 examples in v0.2: desktop form, mobile screen, marketing section. Drives home that compose works for any shape if the kit exists.
holdsComposition breakdown + 0 off-system pill
The "what got used + 0 hardcoded values" panel is the core trust mechanism. Works regardless of shape.
bends"Matches existing recipe" callout
For a project with 17 token versions and active iteration, the recipe library is probably small at first. The callout works but needs a fallback for "no matching recipe yet — this could be the first."
missingTheme target in the compose flow
Writer Handhelds is dark-mode-first per its config (dark mapping is non-trivial). When composing, does the preview render in light, dark, or both? Wireframe shows only one. Refinement: add a theme toggle in the preview panel; or a "render both" mode for projects with multiple themes.
holdsPromotion loop ("used 3+ times → review")
Shape-agnostic. Works as drawn.
missingRound-trip: compose → ship → archaeology catches it
The big-loop diagram in 05 says "compose → ship → archaeology → drift → back." But for a mobile project without HTML-archaeology yet, the loop doesn't actually close. Refinement: the loop diagram should note which ingestion sources are live today vs roadmap.

Rollup · what writer-handhelds taught us

What held up
  • Brand ladder + anchor logic (Step 1 of onboarding)
  • Six-page Figma file structure (shape-agnostic)
  • Promises / doesn't-promise contract
  • Recipes vs L2 distinction
  • Catalog grid + fidelity score format
  • Drift property table + spec auto-draft
  • Org rollup, trend, exec slide
  • Intent input + composition breakdown
  • Promotion loop
What broke or bent
  • Onboarding assumed one brand color → real projects need 8 roles
  • No theme/dark-mode step in onboarding
  • Archetype grid was desktop-web only → no mobile archetypes
  • L2 kit example was dashboard → didn't prove "kit per shape"
  • Archaeology assumed HTML → native mobile needs different ingestion
  • Drift table missing theme dimension
  • Per-PR drift assumed web cadence → mobile is per-release/per-build
  • Compose preview shows only one theme
  • Big-loop diagram doesn't say which ingestion sources are live

Biggest single insight: v0.1 wireframes are dashboard-shape biased. Writer Handhelds exposed that the "archetype kit per shape" thesis is genuinely real — but only one shape (web dashboard) is visible in the wireframes. v0.2 must show at least two shapes side-by-side or the central thesis isn't proven by the artifact. ★
